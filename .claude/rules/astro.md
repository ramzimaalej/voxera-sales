# Astro rules

Guidaro sales-consulting site on Astro 6.x, deployed to **Cloudflare Workers** via the `@astrojs/cloudflare` adapter (`astro.config.mjs`). German-language (DACH); `lang="de"` is the default.

## Render mode
- **Default: static.** Pages render to HTML at build time. Don't add `export const prerender = false` without a reason — per-request rendering on the adapter costs CDN cacheability, and this is a marketing site.
- The `@astrojs/cloudflare` adapter is present so individual routes *can* opt into on-request rendering (forms, dynamic endpoints) when genuinely needed — keep that the exception, not the default.
- If you need a route to render per-request, set `prerender = false` on that route only and document why in an ADR (allocate the number from `../voxera-os/decisions/ADR-REGISTRY.md`). Don't flip the whole site to SSR.

## Components + composition
- Astro components (`.astro`) for HTML structure. No framework needed for static UI.
- Add a UI framework (React/Vue/Svelte) only when interactivity demands it. Don't bring React in to render a static card.
- Component file = one responsibility. Long components → extract.
- Co-locate: page-specific components in `src/components/<page>/` or inline in the `.astro` page file. Reusable ones (`Nav.astro`, `Footer.astro`) stay flat in `src/components/`.
- Shared chrome lives in `src/layouts/Layout.astro` — wrap every page in it rather than re-declaring `<head>`/`<html>` per page.

## Islands (interactive components)
- Default Astro components are 0 KB of JS — ship as static HTML.
- Mark interactivity explicitly with client directives, picking the **most restrictive** that works:
  - `client:visible` — hydrates when the component scrolls into view (use this most of the time).
  - `client:idle` — hydrates after the page is idle (use for non-critical interactivity).
  - `client:load` — hydrates immediately (only for above-the-fold interactivity).
  - `client:only="<framework>"` — render entirely on the client (no SSR HTML); use sparingly, hurts LCP.
- Don't ship a heavy framework runtime for one component. If it's < 100 lines of interaction, write vanilla `<script>` in the `.astro` file.

## Scripts in `.astro` files
- Use `<script>` blocks for client-side JS. Astro bundles + tree-shakes them automatically.
- `<script is:inline>` only for tiny critical-path scripts (e.g., theme detection before paint). Inline scripts skip bundling and aren't deduped.
- Don't write `<script src="..."></script>` pointing at `public/` JS unless it's a vendor blob (analytics) — let Astro bundle.

## Styling
- Single hand-written stylesheet (`src/styles/global.css`) with CSS custom properties for theme tokens (`--navy`, `--blue`, `--accent`, `--font-head`, …). Import it once in `Layout.astro`; don't add a second global sheet. Don't introduce Tailwind/Emotion/styled-components.
- Component-scoped `<style>` blocks are allowed for component-local styling — Astro scopes them automatically.
- Don't override theme tokens with inline `style="..."` — use the variables (`color: var(--accent)`).
- Brand tokens (colors, typography, voice) are documented in this repo's **local Guidaro brand**: `docs/brand/brand-guidelines.md`. Keep the CSS custom properties in `global.css` in sync with it; the brand file is the source of truth. Guidaro is its own brand — never pull tokens or voice rules from another repo's brand.

## Content collections
- If structured/listing content (case studies, blog posts) grows beyond a couple of hand-written pages, move it to `src/content/` with a schema in `src/content/config.ts` — type-safe and validated at build time.
- Don't fetch markdown from `public/` at runtime when content collections work — you lose type safety + dev-time validation.

## Copy + language
- The site is German-language. Page copy lives directly in the `.astro` pages (`branchen`, `leistungen`, `ueber-uns`, `kontakt`, `impressum`) — there is no separate string bundle to keep in sync.
- Before merging a copy change, check it against `docs/brand/brand-guidelines.md` (voice, forbidden words, CTA patterns). Match the established German tone — formal "Sie", no anglicisms the brand guide forbids.
- If a second locale is ever added, introduce a string bundle + helper at that point and document the decision in an ADR — don't pre-build i18n the single-locale site doesn't need.

## Assets
- Static assets (favicons, illustrations, logos) live in `public/` and are served as-is at the root path (`/favicon.svg`, `/img/...`).
- Don't import from `public/` in components — `public/` is for assets served verbatim. Images that should be optimized go through Astro's image pipeline, not `public/`.

## Build + deploy output
- `dist/` is generated by `astro build`. Never edit by hand. Never commit `dist/` to git.
- Cloudflare serves `dist/` via the Worker entrypoint configured in `wrangler.jsonc` (`assets.directory` → `./dist`). Deploy with `npm run deploy` (`astro build && wrangler pages deploy dist`).
- `worker-configuration.d.ts` / generated types come from `wrangler types` (`npm run generate-types`) — regenerate rather than hand-editing.

## Quality gates
- `npm run lint` — ESLint (flat config, `eslint-plugin-astro` + `jsx-a11y`). Keep accessibility warnings down, don't just raise `--max-warnings`.
- `npm run typecheck` — `astro check` (TypeScript strict via `astro/tsconfigs/strict`). Must pass clean before merge.
- `npm run build` — must succeed; a broken build blocks deploy.

## Don't
- Don't add a state management library (Redux/Zustand/Jotai). The site is mostly static; islands manage local state with `useState` or vanilla.
- Don't add a CMS dependency for content that's stable enough to live in `src/content/` (or directly in pages).
- Don't import from `public/` in components — `public/` is for static assets served as-is.
- Don't disable Astro's image optimization for a quick perf win; the optimization is the perf win.
- Don't flip the whole site to SSR or remove the static default to make one route dynamic — scope `prerender = false` to that route.
- Don't reference another repo's brand. Guidaro's brand is local (`docs/brand/brand-guidelines.md`); the Voxera CRM brand does not apply here.
